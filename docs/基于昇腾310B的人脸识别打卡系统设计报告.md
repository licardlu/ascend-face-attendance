# 基于昇腾310B的人脸识别打卡系统设计报告

## 1. 项目概述

### 1.1 项目背景

本项目来源于课程设计任务，基础代码参考教师提供的 `Ascend310-main/samples/case1` 人脸打卡例程。项目面向教室、实验室等小型考勤场景，希望利用 OrangePi AI Pro / 昇腾 310B 边缘 AI 平台完成本地摄像头采集、本地 NPU 推理、本地数据库记录和网页端展示，减少人工签到中的代签、漏签和统计繁琐问题。

系统采用 Flask 提供 Web 服务，OpenCV 完成摄像头采集与图像处理，AscendCL/CANN 运行人脸检测与人脸特征提取模型，SQLite 保存用户和考勤记录。网页端提供用户注册、实时考勤、用户管理、考勤记录展示等功能。

### 1.2 建设目标

根据课程设计计划书，本项目总体目标如下：

1. 完成昇腾 310B 开发环境配置，保证 CANN 8.3.RC1 与教师例程环境一致。
2. 跑通教师 case1 人脸打卡样例，包括模型转换、摄像头访问和网页端展示。
3. 在原始识别打卡流程上完善用户注册、考勤记录、重复打卡控制和异常状态提示。
4. 优化前端页面，使系统更适合课程设计演示和答辩展示。
5. 为后续活体检测、考勤统计和数据导出预留清晰扩展位置。

### 1.3 设计原则

- 先保证基础闭环可运行，再逐步扩展创新功能。
- 注册、识别、考勤三个业务流程保持边界清晰。
- 端侧采集、端侧推理、端侧记录，尽量减少对云端服务的依赖。
- 前端页面以可操作、可演示、可理解为核心。
- 数据库逻辑集中处理时间、去重和删除规则，减少前后端重复判断。

## 2. 需求分析

### 2.1 功能需求

| 功能模块 | 需求说明 | 当前实现状态 |
| --- | --- | --- |
| 用户注册 | 通过上传图片或摄像头抓拍注册用户人脸特征 | 已实现 |
| 用户管理 | 展示已注册用户、编辑姓名、删除用户 | 已实现 |
| 批量删除 | 勾选多个错误注册用户后一次性删除 | 已实现 |
| 人脸识别考勤 | 摄像头实时识别已注册用户并生成考勤记录 | 已实现 |
| 未注册提示 | 识别到未注册人员时只提示，不自动注册 | 已实现 |
| 重复打卡控制 | 同一用户当天只记录第一次考勤 | 已实现 |
| 北京时间校准 | 注册时间和考勤时间统一使用北京时间 | 已实现 |
| 前端展示优化 | 首页、用户管理页、考勤页统一视觉风格 | 已实现 |
| 活体检测 | 通过眨眼、摇头、张嘴或连续帧变化判断真人 | 计划扩展 |
| 考勤统计 | 按日期、人员、状态统计出勤情况 | 计划扩展 |
| 数据导出 | 导出 CSV 或 Excel 考勤数据 | 计划扩展 |

### 2.2 非功能需求

1. **稳定性**：系统应能在板端长时间运行，摄像头不可用或模型加载失败时给出明确提示。
2. **实时性**：摄像头画面持续输出，自动考勤按固定间隔进行识别，避免过高频率导致板端负载过大。
3. **可维护性**：核心模块按职责划分，便于后续添加活体检测、统计导出等功能。
4. **安全性**：上传文件名做路径清理，避免路径穿越；人脸数据存储在本地，不上传云端。
5. **演示性**：网页端展示注册、识别、未注册、重复考勤等状态，便于课堂答辩演示。

## 3. 总体架构设计

### 3.1 系统架构

系统采用浏览器/服务器结构，浏览器负责页面交互和状态展示，Flask 后端负责业务接口、摄像头视频流和数据访问，昇腾 310B 负责模型推理。

```text
浏览器前端
  |-- 首页
  |-- 用户管理页
  |-- 实时考勤页
        |
        v
Flask Web 服务
  |-- 页面路由
  |-- 用户 API
  |-- 考勤 API
  |-- 摄像头状态 API
  |-- 视频流接口
        |
        +----------------+
        |                |
        v                v
SQLite 数据库       摄像头与推理模块
  |-- users 表       |-- OpenCV 摄像头采集
  |-- attendance 表  |-- 人脸检测 OM 模型
                    |-- 人脸识别 OM 模型
                    |-- 特征比对与状态输出
```

### 3.2 技术选型

| 层次 | 技术 | 作用 |
| --- | --- | --- |
| 硬件平台 | OrangePi AI Pro / 昇腾 310B | 边缘 AI 推理 |
| AI 环境 | CANN 8.3.RC1、AscendCL | OM 模型加载与推理 |
| 图像处理 | OpenCV | 摄像头采集、图像读写、帧编码 |
| 后端服务 | Python、Flask | 页面路由和 REST API |
| 数据存储 | SQLite | 用户信息和考勤记录保存 |
| 前端页面 | HTML、Bootstrap、CSS、JavaScript | 页面交互和状态展示 |
| 测试验证 | unittest | 数据库、摄像头逻辑、接口辅助函数和模板结构测试 |

## 4. 模块设计

### 4.1 Web 服务模块 `app.py`

`app.py` 是系统入口，负责 Flask 应用初始化、页面路由和 REST API。

主要职责：

- 渲染首页、用户管理页和考勤记录页。
- 提供用户增删改查接口。
- 提供批量删除用户接口。
- 提供摄像头抓拍接口。
- 提供摄像头识别状态接口。
- 提供考勤记录查询接口。
- 在 Windows 或无硬件依赖环境下允许网页预览启动。

主要接口：

| 接口 | 方法 | 功能 |
| --- | --- | --- |
| `/` | GET | 首页 |
| `/users_page` | GET | 用户管理页面 |
| `/attendance_page` | GET | 考勤页面 |
| `/api/users` | GET | 查询用户列表 |
| `/api/users` | POST | 注册用户 |
| `/api/users/<user_id>` | PUT | 修改用户姓名 |
| `/api/users/<user_id>` | DELETE | 删除单个用户 |
| `/api/users/batch_delete` | POST | 批量删除用户 |
| `/api/camera/capture` | POST | 从设备摄像头抓拍注册图片 |
| `/api/camera/status` | GET | 查询最近一次摄像头识别状态 |
| `/api/attendance` | GET | 查询当天考勤记录 |
| `/video_feed` | GET | 输出 MJPEG 摄像头视频流 |

### 4.2 摄像头与自动考勤模块 `camera.py`

`camera.py` 中的 `VideoCamera` 类负责摄像头生命周期管理和自动考勤识别。

主要职责：

- 打开本地摄像头设备。
- 后台线程持续读取摄像头帧。
- 输出 MJPEG 视频流给前端显示。
- 按固定间隔执行人脸检测和识别。
- 根据识别结果设置前端可轮询的状态。
- 对未注册人员只提示 `未注册`，不写入用户表或考勤表。
- 对已注册用户调用数据库层考勤写入逻辑，自动区分首次考勤和重复考勤。

当前状态类型包括：

| 状态 | 含义 | 页面表现 |
| --- | --- | --- |
| `idle` | 等待识别 | 不显示提示层 |
| `no_face` | 未检测到人脸 | 可用于调试提示 |
| `unregistered` | 检测到未注册人脸 | 显示“未注册” |
| `recognized` | 识别到已注册用户 | 显示 Welcome 或已经考勤 |

### 4.3 昇腾推理模块 `ascend_inference.py`

`ascend_inference.py` 对 AscendCL 推理流程进行封装，主要包括设备初始化、模型加载、输入输出内存管理和推理执行。

核心类：

- `AscendSystem`：负责 ACL 初始化、设备上下文和 stream 管理。
- `AscendModel`：负责 `.om` 模型加载、输入输出 buffer 管理和推理执行。
- `FaceSystem`：组合人脸检测模型和人脸识别模型，向业务层提供 `detect()` 和 `get_embedding()`。

人脸识别流程：

```text
摄像头帧/上传图片
  -> OpenCV 读取或解码
  -> 人脸检测模型检测人脸框
  -> 选取最大人脸区域
  -> 人脸识别模型提取 512 维特征
  -> 与数据库特征做余弦相似度比对
  -> 根据阈值判断是否为已注册用户
```

### 4.4 数据库模块 `database.py`

`database.py` 负责 SQLite 数据库初始化和所有持久化操作。当前设计将时间校准、重复考勤判断、用户删除级联清理等规则集中放在数据库层，保证自动考勤和手动打卡使用同一套数据规则。

主要职责：

- 初始化 `users` 和 `attendance` 表。
- 写入用户注册信息和人脸特征。
- 查询用户列表。
- 单个或批量删除用户，并同步删除对应考勤记录。
- 以北京时间写入注册时间和考勤时间。
- 同一用户同一天只保留第一次考勤记录。
- 查询当天每个用户的首次考勤记录。

### 4.5 前端页面模块

前端页面位于 `src/case1/templates` 和 `src/case1/static/css`。

| 文件 | 作用 |
| --- | --- |
| `templates/index.html` | 首页，展示项目定位和入口 |
| `templates/users.html` | 用户注册、摄像头抓拍、用户列表、批量删除 |
| `templates/attendance.html` | 实时视频流、识别状态提示、当天考勤记录 |
| `static/css/app.css` | 统一页面视觉风格和组件样式 |

前端设计方向采用“考勤终端/管理面板”风格，突出实时摄像头、用户库维护、今日考勤和规则提示，不做营销式页面。

## 5. 数据库设计

### 5.1 用户表 `users`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | 数据库内部用户 ID |
| `name` | TEXT NOT NULL | 用户姓名 |
| `embedding` | BLOB | 人脸特征向量 |
| `avatar` | TEXT | 头像图片文件名 |
| `created_at` | TIMESTAMP | 用户注册时间，北京时间 |

说明：页面上显示的“序号”不直接使用数据库 ID，而是使用前端列表索引 `index + 1`。这样删除用户后，页面序号仍能从 1 开始连续显示，数据库 ID 则继续作为内部稳定标识使用。

### 5.2 考勤表 `attendance`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | 考勤记录 ID |
| `user_id` | INTEGER | 对应用户 ID |
| `timestamp` | TIMESTAMP | 考勤时间，北京时间 |
| `type` | TEXT | 考勤类型，如 `camera_auto` 或 `manual` |
| `image_path` | TEXT | 考勤抓拍图片文件名 |

考勤写入规则：

1. 写入前先查询该用户当天是否已有记录。
2. 如果没有记录，则插入新考勤并返回 `created=True`。
3. 如果已有记录，则不重复插入，返回第一次考勤记录并标记 `created=False`。
4. 前端根据返回状态显示 `Welcome` 或 `已经考勤`。

## 6. 关键业务流程设计

### 6.1 用户注册流程

```text
进入用户管理页
  -> 输入姓名
  -> 上传图片或摄像头抓拍
  -> POST /api/users
  -> 后端读取图像
  -> 人脸检测
  -> 裁剪最大人脸
  -> 提取人脸特征
  -> 保存头像和特征向量
  -> 写入 users 表
  -> 前端刷新用户列表
```

设计要点：

- 注册只能由用户管理页面主动触发。
- 考勤识别流程不会自动创建用户。
- 用户头像用于前端列表展示，特征向量用于识别比对。

### 6.2 自动考勤流程

```text
启动 Flask 服务
  -> 初始化数据库
  -> 初始化 FaceSystem
  -> 初始化摄像头线程
  -> 持续读取摄像头帧
  -> 定时执行 process_attendance
  -> 检测人脸并提取特征
  -> 与 users 表中已注册特征比对
        |-- 未匹配：设置状态为 unregistered，不入库
        |-- 已匹配：调用 add_attendance
                |-- 当天首次：写入考勤，显示 Welcome
                |-- 已考勤：不重复写入，显示已经考勤
```

设计要点：

- 自动考勤只服务于识别和考勤，不承担注册职责。
- 未注册人员不进入用户表，也不进入考勤表。
- 同一用户当天只保留第一次考勤时间。

### 6.3 批量删除流程

```text
用户管理页加载用户列表
  -> 勾选多个用户
  -> 点击批量删除
  -> POST /api/users/batch_delete
  -> 后端校验 ids
  -> 删除 attendance 中相关记录
  -> 删除 users 中相关记录
  -> 返回删除数量
  -> 前端刷新用户列表
```

设计要点：

- 批量删除接口只接收 ID 列表。
- 后端会过滤非法 ID，避免异常输入导致 SQL 错误。
- 删除用户时同步删除该用户考勤记录，避免数据残留。

### 6.4 摄像头状态轮询流程

```text
考勤页面打开
  -> 每 1 秒请求 /api/camera/status
  -> 获取最近一次识别状态
  -> 根据 sequence 判断是否为新状态
  -> 显示对应提示层
  -> 必要时刷新今日考勤表
```

设计要点：

- 前端不再通过考勤列表数量变化猜测是否打卡。
- 状态接口能明确区分未注册、首次考勤和重复考勤。
- `sequence` 字段用于避免同一状态反复触发动画。

## 10. 后续扩展设计

### 10.1 活体检测模块

计划新增 `liveness` 模块，在人脸识别前增加活体判断。可选方案包括：

1. **动作活体**：根据眨眼、摇头、张嘴等动作判断是否为真人。
2. **连续帧变化**：比较连续帧中人脸区域、姿态或关键点变化。
3. **交互式提示**：前端提示“请眨眼”“请摇头”“请靠近摄像头”，后端验证动作结果。

活体检测通过后才进入特征比对和考勤写入；活体失败时显示“活体检测失败”，不写入考勤记录。

### 10.2 考勤统计模块

计划在现有考勤记录基础上扩展统计页面：

- 今日已签到人数。
- 今日未签到人数。
- 最近打卡记录。
- 按日期查询历史考勤。
- 按用户查看个人考勤情况。
- 后续可扩展迟到、缺勤状态。

### 10.3 数据导出模块

计划新增 CSV 或 Excel 导出接口，导出字段包括：

- 日期。
- 姓名。
- 用户 ID 或学号。
- 首次打卡时间。
- 考勤状态。
- 打卡方式。

导出功能用于课程验收、教师查看和后续数据分析。

## 11. 风险与应对

| 风险 | 表现 | 应对方案 |
| --- | --- | --- |
| 板端环境差异 | CANN、ACL、模型路径不一致导致启动失败 | 固化部署步骤，记录环境版本，启动前检查模型文件 |
| 摄像头设备差异 | `/dev/video0` 或 `/dev/video2` 编号不一致 | 将摄像头编号配置化，启动时输出错误提示 |
| 识别阈值不合适 | 误识别或无法识别 | 保留 `RECOGNITION_THRESHOLD` 配置，按演示环境调参 |
| 光照影响 | 人脸检测不稳定 | 演示时保证正面光照，注册清晰头像 |
| 重复打卡误判 | 跨天或时区导致记录错误 | 后端统一使用北京时间日期判断 |
| 活体检测复杂度过高 | 影响实时性或难以稳定演示 | 优先实现轻量级可解释方案 |

## 12. 总结

本系统在教师提供的昇腾 310B 人脸打卡样例基础上，围绕课程设计计划书提出的“真实、可靠、可管理、可展示”目标进行了系统化设计和功能完善。当前版本已经完成用户注册、人脸识别考勤、未注册提示、重复打卡控制、北京时间校准、用户批量删除和前端页面美化等核心功能，形成了可演示的人脸考勤闭环。

后续可继续按照计划书推进活体检测、考勤统计和数据导出功能，使系统从基础识别打卡进一步扩展为具备异常拦截、统计分析和课程答辩展示能力的完整边缘 AI 考勤系统。
